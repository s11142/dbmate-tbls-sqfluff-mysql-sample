## 1. 総評

**結論: ベースとしては優秀だが、そのまま本業投入は厳しい。一部修正が必須。**

このリポジトリは dbmate + SQLFluff + tbls という組み合わせでマイグレーション管理の「型」を作ろうとしている点で方向性は正しく、個人開発やプロトタイプとしては十分な品質がある。しかし、**接続情報のハードコード**、**破壊的変更の検知が warning 止まり**、**tbls の latest 利用**、**host network 依存**など、複数人チームでの業務利用においてはセキュリティ・再現性・運用安全性の面でそのまま使うと事故につながるリスクがある。改修ポイントは明確であり、12 件の Issue 対応で業務投入レベルに引き上げられる。

---

## 2. 良い点

- **ツール選定が堅実**: dbmate（軽量・Go 製・SQL ベース）、SQLFluff（Python 製で MySQL 方言対応）、tbls（ER 図 + Markdown 自動生成）の組み合わせは、ORM 非依存のマイグレーション管理として実務的に妥当
- **migration の up/down が両方記述されている**: サンプルの migration ファイルともに down セクションが書かれており、rollback を意識した設計になっている
- **CI で往復テストがある**: `migration-test.yml` で up → 全 rollback → 再 up の往復テストを実施しており、migration の可逆性を担保する仕組みがある
- **Makefile でコマンドが統一されている**: 開発者が覚えるべきコマンドが `make xxx` に集約されており、オンボーディングコストが低い
- **README が充実している**: 目次、コマンド一覧、ディレクトリ構成、運用ガイド、トラブルシューティングまで網羅されており、初見でも使い始められる
- **docker-compose で healthcheck を設定している**: MySQL の起動完了を待ってから dbmate を実行する設計になっている
- **SQLFluff のバージョンが固定されている**: `sqlfluff/sqlfluff:3.3.0` でバージョンが固定されており、lint の再現性が確保されている
- **危険コマンドチェックがある**: `lint.yml` で `DROP TABLE` / `TRUNCATE` 等を migrate:up セクション内で検出する仕組みがある

## 3. 懸念点・問題点

### 優先度: 高（業務投入前に必ず修正）

1. **接続情報（パスワード・DSN）が全箇所にハードコードされている**
   - `docker-compose.yml`、`Makefile`、`.tbls.yml`、CI ワークフロー 2 ファイル、`README.md` の計 6 箇所以上に平文パスワードが記載
   - `.env` ファイル + `.env.example` パターンに移行すべき

2. **破壊的 SQL 検知が `::warning` で素通りする**
   - CI が green のまま通過するため、レビュアーが warning に気づかなければ破壊的変更がマージされる
   - `exit 1` にして fail させるべき

3. **tbls が `latest` タグで実行されている**
   - バージョンアップで出力形式が変わると差分が大量発生する
   - バージョンを固定すべき

4. **`--network host` に依存している**
   - macOS / Windows WSL2 では動作しない可能性がある
   - Docker network 経由で接続する構成に変更すべき

5. **`schema_migrations` テーブルが docs に含まれている**
   - `.tbls.yml` の `exclude` 設定で除外すべき

### 優先度: 中（早期に対応すべき）

6. **ポート 3306 の固定利用でプロジェクト間競合が発生する**
7. **migration 設計ガイドラインが存在しない**
8. **`db/schema.sql` が存在しない（source of truth が不明確）**
9. **docs 自動コミットが main ブランチに直接 push される**
10. **dbmate のバージョンがメジャーバージョン指定のみ**

### 優先度: 低（余裕があれば対応）

11. **SQLFluff 設定ファイルがコンテナにマウントされていない**
12. **CI ワークフローのトリガーパスに設定ファイルの変更が含まれていない**

## 本業投入の可否（まとめ）

このリポジトリは dbmate + SQLFluff + tbls によるマイグレーション管理のテンプレートとして方向性は正しく、構成の 7〜8 割はそのまま活用できる。特に CI での往復テスト、Makefile による操作の統一、README の充実度は評価できる。しかし、**接続情報のハードコード**（Issue 1）、**破壊的変更の素通り**（Issue 2）、**SQLFluff 設定の未適用**（Issue 10）は品質ゲートとしての信頼性を根本から損なうため、最低限この 3 件は業務投入前に修正が必須である。加えて、tbls のバージョン固定（Issue 3）と host network 依存の解消（Issue 4）もチーム開発では早期に対応すべき。全 12 件の Issue を優先度順に対応すれば、複数人チームでの本業利用に十分耐えうるマイグレーション管理基盤になる。
